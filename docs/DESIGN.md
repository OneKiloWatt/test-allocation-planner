# DESIGN

## UIの基本方針
- スマホファースト
- 1画面1目的
- その時点で `今日やること` または `今やるべきこと` を1つ強く出す
- 分析画面より、行動に移れる画面を優先する

## 利用開始の考え方
- MVPでは `匿名で試す` を許可する
- ただし、匿名利用は端末依存であることを隠さない
- ログインは最初から強制せず、必要になったタイミングで提案する

## ホームの考え方
ホームは固定トップではなく、状態別トップにする。

- テスト未作成:
  - `テストを作成する`
- 配分未作成:
  - `目標と予定を入れて配分案を作る`
- テスト前:
  - 上部に `中間テストまであと6日` のような状況表示
  - 主役は `今日やること`
  - その下に進捗サマリー
- テスト後で結果未入力:
  - `結果を入力する`

## 今日やることの考え方
- テストが登録済みで、配分が確定していて、今日がブロック日でない場合に表示する
- 出すのは `今日のおすすめ科目` と `目安時間`
- ここからそのまま `進捗登録` へ進めるようにする

例:
- `今日は数学 60分、理科 30分`
- `数学は残り6時間`
- `記録する`

## ホーム上部の状況表示
- テスト前は、上部に `2学期中間テストまであと6日` のような細めの状況カードを置く
- ここで見せるのは次まで
  - テスト名
  - 開始日
  - 残り日数
- 大きすぎるヒーローではなく、文脈を伝えるためのヘッダーに留める

## 入力の考え方
- 1回で全部入れさせない
- `テスト作成`
- `目標点数`
- `日程/予定`
- `進捗登録`
- `結果入力`
  の3回に分ける

## 目標点数と日程を分ける理由
- `どこまで取りたいか` と `いつ勉強できるか` は別の思考
- 同じ画面に置くと入力量が多く見える
- 先に目標を決め、その後に日程化モードを選ぶ方が理解しやすい

## 予定入力UIの考え方
- 予定入力では最初に `自動で組む` か `手動で組む` かを選べるようにする
- デフォルトは `自動で組む`
- 手動を選んだ場合、生活パターン入力はスキップできる

### 自動で組む
- 最初に見せるもの:
  - 部活がある平日
  - 部活がない平日
  - 土日
  - 部活がある曜日
- 詳細設定として折りたたむもの:
  - テスト1週間前は部活なし扱い
  - 勉強開始日を変更
- 補助表示:
  - `通常は今日から計画します`
  - `開始日: 今日 変更`
- MVPでは見せないもの:
  - その日だけ勉強時間を上書きする設定
  - 日ごとの例外設定

### 手動で組む
- 日ごとに `この日にこの科目をやる` を決める
- Googleカレンダーのような時間単位編集まではやらない
- まずは `日ごとの学習テーマを置くカレンダー` を目指す

## 表示の考え方
- グラフは最小限
- まず見せるのは `前回配分` と `今回の配分サマリー`
- 長期推移は後回し

## 振り返り画面の考え方
- 振り返りは `ログ一覧` ではなく `テストごとの結果カード` で見せる
- 1カードに入れるのは次まで
  - テスト名
  - 学年/学期
  - 科目ごとの勉強時間
  - 目標点
  - 実際の点数
  - `達成 / 未達`
  - 一言まとめ
- ユーザーが見たいのは生ログではなく、`この時は足りていたか` `次回にどう活かせるか`
- 次回テスト作成時に `この結果を引き継ぐ` 導線をカードから自然につなげる

## 重要なUX方針
- テスト終了後に結果入力を探させない
- 前回比較は自然に差し込む
- 修正導線を必ず置く
- 終わった後は `次回へつながる感じ` を出す
- 勉強日ならホームに `今日やること` を主役で出す
- 進捗登録は `科目 + 時間` だけで終わる軽さにする
- 記録後すぐに `残り時間` を返す
- 自動設定の主入力と、詳細設定/特殊ケースを分けて見せる
- 手動で調整した行は `手動` と明示し、再計算で保持されることを伝える
- 匿名利用の制約は、保存後の軽いポップアップで知らせる

## 文言ルール
- 使う:
  - `配分サマリー`
  - `参考プラン`
  - `前回より理科を少し増やす案`
- 使わない:
  - `最適解`
  - `確実に上がる`
  - `危険`

## 色と見た目の方針
- 学習管理アプリらしく、焦らせすぎない
- 赤で煽らない
- 未達表示は責める色より、中立な注意色

## カラーパレット
- ベース: 白に近いクリーム（B2トーン）
  - 学習帳・ノートのような親しみやすさ
  - 男女・年齢を問わず受け入れやすい
- アクセント: テラコッタ（くすんだ暖色オレンジ、HSL 22, 65%, 52%）
  - 色鉛筆・クラフト紙のような文房具感
  - 彩度を落とすことで警告色・alert-orangeと区別する
- 使わない: 赤・高彩度オレンジ（警告・締め切り感が出るため）

## コンポーネント語彙
- ボトムシート: 短い入力・折りたたみ設定のデフォルト
  - 例: 日付タップ→科目追加、詳細設定、進捗記録
- ページ遷移: 入力項目が多くスクロールが必要なフォーム
  - 例: テスト作成フォーム
- モーダル: 破壊的操作の確認のみ
  - 例: テスト削除の確認
- ルール: 「短い入力はボトムシート、長いフォームはページ遷移、モーダルは破壊的操作専用」

## デザイントークン完了条件
- MVP のカラートークン最小セットとして `background` `surface` `text` `muted` `border` `primary` `primary-foreground` `destructive` `warning` を定義する
- 状態色として `active tab` `inactive tab` `muted text` `success` `caution` `destructive` を定義し、文言やアイコンの意味とセットで使い分ける
- ダークモードは MVP の対象外とし、ライトテーマの読みやすさと一貫性を先に固める
- トークンは Tailwind の custom color と CSS 変数の両方で参照できる状態を完了条件とする

## デザイントークン運用ルール
- source of truth は `globals.css` の CSS 変数とする
- Tailwind から使う値は `theme.extend` 経由で CSS 変数を参照し、アプリコード側で直接色値を再定義しない
- カラー、余白、寸法ともに「先に CSS 変数を定義し、その別名を Tailwind に公開する」順で運用する
- 一時対応であっても `tailwind.config` に16進色や固定寸法値を直書きして二重定義しない
- コンポーネント個別 CSS や JSX の `style` 属性へ色値を直書きしない。例外が必要な場合は先にトークン化する

### 命名規則
- CSS 変数は `--color-*` `--size-*` `--space-*` の接頭辞で用途を明示する
- Tailwind 側の公開名は CSS 変数名の接頭辞を外した kebab-case にそろえる
- 対応ルール:
  - `--color-background` → `background`
  - `--color-primary-foreground` → `primary-foreground`
  - `--space-screen-x` → `px-screen`
  - `--size-header-mobile` → `h-header-mobile`
- 状態トークンも同じ対応規則に統一し、実装ごとの別名を増やさない

### 二重定義防止ルール
- `tailwind.config` の `extend.colors` `extend.spacing` `extend.height` `extend.maxWidth` には CSS 変数参照だけを書く
- Figma や仕様書から値を更新するときは、`globals.css` の変数値を先に直し、Tailwind 側は参照名の追加・削除が必要な場合にだけ更新する
- 同じ意味のトークンを `surface` と `cardBg` のように複数名で増やさない
- レビュー時は「CSS 変数未定義の Tailwind token」「Tailwind 未公開の独自 CSS 変数」「直書き色値」の3点を確認対象にする

## レイアウト寸法トークン
- スマホファーストの基準幅は最大幅 390px とする
- 画面レイアウトの基準値は次を採用する
  - ヘッダー高（スマホ基準）: `56px`
  - ボトムナビ高: `64px`
  - ボトムナビ下部余白: `env(safe-area-inset-bottom)` を別加算する
  - 画面左右余白: `16px`
  - コンテンツ最大幅: `390px`
  - カード間隔: `12px`
  - セクション間隔: `24px`
- ボトムナビを持つ画面では、主コンテンツの下側余白に `64px + env(safe-area-inset-bottom)` 以上を確保する
- カード内余白の基本値は左右上下とも `16px` を起点とし、情報量が多い場合のみ縮小を検討する
- フォーム画面、ホーム画面、レビュー画面で寸法ルールを変えすぎず、上記トークンの組み合わせで収める

### 寸法トークンの公開例

```css
/* globals.css */
:root {
  --size-header-mobile: 56px;
  --size-bottom-nav: 64px;
  --space-screen-x: 16px;
  --size-content-max: 390px;
  --space-card-gap: 12px;
  --space-section-gap: 24px;
}
```

```ts
// tailwind.config.ts
export default {
  theme: {
    extend: {
      spacing: {
        "px-screen": "var(--space-screen-x)",
        "card-gap": "var(--space-card-gap)",
        "section-gap": "var(--space-section-gap)",
      },
      height: {
        "header-mobile": "var(--size-header-mobile)",
        "bottom-nav": "var(--size-bottom-nav)",
      },
      maxWidth: {
        content: "var(--size-content-max)",
      },
    },
  },
}
```

### 役割ごとの色の考え方
- `background`: 画面全体の地色。白すぎず、ノート感のあるクリーム
- `surface`: カードやシートの面
- `text`: 本文と主要ラベル
- `muted`: 補助背景や弱い区切り
- `border`: 入力枠線とカード境界
- `primary`: テラコッタ基調の主要CTA
- `primary-foreground`: `primary` 上の文字色
- `warning`: 中立な注意。締切煽りに見えない黄土寄り
- `destructive`: 削除など破壊的操作専用

### 状態色

| 状態 | 用途 |
| --- | --- |
| `active tab` | 現在地のタブ。`primary` 系を使う |
| `inactive tab` | 非選択タブ。主張しない茶灰色 |
| `muted text` | 補助説明、日付、注釈 |
| `success` | 記録完了、達成表示 |
| `caution` | 未入力、要確認、軽い注意 |
| `destructive` | 削除、取り消し不能操作、重大エラー |

### Tailwind custom color 定義例

```ts
// tailwind.config.ts
export default {
  theme: {
    extend: {
      colors: {
        background: "#F7F1E8",
        surface: "#FFF9F2",
        text: "#3F342C",
        muted: "#E8DDD0",
        border: "#D7C7B4",
        primary: "#D67A52",
        "primary-foreground": "#FFF7F0",
        warning: "#D9A441",
        destructive: "#B85C38",
        "active-tab": "#C96E46",
        "inactive-tab": "#8F7A69",
        "muted-text": "#7D6C5F",
        success: "#6F9B6E",
        caution: "#C6923A",
      },
    },
  },
}
```

- `background` と `surface` の差は弱めにして、まぶしさを抑える
- `primary` はボタン・強調ラベルまでに留め、画面全体を暖色で埋めない
- `destructive` は赤寄りにしすぎず、通常導線の `primary` と明確に区別する
