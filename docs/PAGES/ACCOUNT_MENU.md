# ACCOUNT_MENU

## 目的
- アカウント関連操作（ログアウト・退会）と法的文書への導線を提供する

## 配置
- ヘッダー右上のアカウントアイコン押下で開く `DropdownMenu`
- 全画面共通の `AppHeader` / `Layout` に置く（画面単位で重複実装しない）

## ログインユーザーのメニュー項目・順番

| 順 | 表示文言 | 操作 |
|---|---|---|
| 1 | ログアウトする | ログアウト処理 |
| 2 | 利用規約 | `/terms` へ遷移 |
| 3 | プライバシーポリシー | `/privacy` へ遷移 |
| 4 | アカウントを削除する | 退会フローへ（赤色表示） |

## ゲストユーザーのメニュー項目・順番

| 順 | 表示文言 | 操作 |
|---|---|---|
| - | ゲストで利用中（非操作テキスト） | — |
| 1 | アカウントを作成する | 登録ページへ遷移 |
| 2 | 利用規約 | `/terms` へ遷移 |
| 3 | プライバシーポリシー | `/privacy` へ遷移 |
| 4 | この端末のデータを消す | ゲストデータ削除フローへ（赤色表示） |

- ゲストユーザーには `ログアウトする` と `アカウントを削除する` は表示しない

## UI方針
- 文言はすべて動詞付き（「ログアウトする」「アカウントを削除する」）
- 危険操作（アカウント削除・データ消去）のみ赤色、他は通常色
- メニュー見出しは「アカウント」
- コンポーネント: `DropdownMenu` + 退会確認は `AlertDialog`（直接実行にしない）

## 退会フロー（ログインユーザー）

### ステップ1: 補助導線の提示
- 文言: `アカウントを消さなくても、この端末ではそのまま使えます`
- 選択肢: `ログアウトする` / `それでも削除する`
- 目的: 勢いでの削除を防ぐ

### ステップ2: 確認画面
- 説明文: `アカウントを削除すると、学習記録と計画は元に戻せません。`
- 主ボタン: `アカウントを残す`
- サブボタン: `削除する`（赤）
- 理由入力: 任意アンケート（「サービス改善のために参考にさせていただきます」と目的を明示する）

### ステップ3: 再認証
- パスワード再入力 / magic link 再確認 / OAuth 再認可のいずれかを要求する

### ステップ4: サーバーサイド削除処理
- Next.js API Route または Supabase Edge Function で実行（`service_role` 権限が必要）
- 削除順: `exams`（user_id 全行）→ `progress_logs`（user_id 全行）→ `auth.users` → global sign out
- 理由: 先に認証を消すと DB クリーンアップ権限を失うため
- 失敗時: 完全ロールバックより「失敗地点の明確化」を優先する
  - DB 削除済み・認証削除失敗の場合は「ログインは残るがデータは空」の状態として再実行可能にする
  - 逆（認証削除済み・DB 残存）は避ける

## ゲストデータ削除フロー

### 確認画面
- 説明文: `この端末に保存されている学習記録が削除されます。元に戻せません。`
- 主ボタン: `残す`
- サブボタン: `削除する`（赤）

### 削除対象（localStorage）
- 学習計画データ、ゲスト ID、同期フラグ、下書き、キャッシュ、最終更新時刻、migration 用キー

### 注意
- `localStorage` 削除はそのブラウザ端末にしか効かないため、UI 文言に「この端末上のデータ」と明示する

## 利用規約・プライバシーポリシー表示

- 別ページ（`pages/terms.tsx` / `pages/privacy.tsx`）として実装
- メニューから通常遷移、ヘッダーに「閉じる / 戻る」
- モーダル不採用の理由: 長文閲覧・URL 共有・将来の改定管理と相性が悪い
- コンテンツ形式: MVP は静的 TSX/HTML、頻繁な改定が生じたら MDX 化を検討
- 記載必須事項: 取得目的・保存期間・第三者提供の有無・削除方法を平易な文言で明記

## 未成年への配慮（プライバシーポリシー記載要件）
- 18歳未満の方は保護者の同意・確認をお願いする旨を記載（TERMS・PRIVACY と統一）
- バックアップ内データの即時物理削除が困難な場合は retention 期間を明示する
- 年齢閾値と保護者同意の具体文言は法務確認を推奨

## セキュリティ要件（実装メモ）
- ログアウト順: `UI ローディング化` → `signOut()` → `Zustand リセット` → `localStorage の auth 関連キーのみ削除` → トップへ遷移
- ログアウトで学習データ（localStorage）は消さない（auth 関連キーのみ対象）
- `onAuthStateChange` と手動 Zustand リセットの二重実行に注意
- JWT 有効期限は短めに設定する
- 退会時は global sign out で全セッション失効させる
- 認証必須ページはキャッシュ制御を入れ、ログアウト後のブラウザバックでデータが見えないようにする

## ゲストユーザー判定
- `authMode: 'guest' | 'authenticated'` を Zustand に持つ
- 初期化時に Supabase session と localStorage の存在から決定する
- `isAnonymous` は Supabase auth の匿名ログイン概念と混同しないよう `guest` と命名する

## やらないこと
- 退会理由の入力を必須にすること
- メニュー内に利用規約・プライバシーポリシーの本文を展開すること
- ゲストユーザーに「ログアウトする」や「アカウントを削除する」を表示すること
