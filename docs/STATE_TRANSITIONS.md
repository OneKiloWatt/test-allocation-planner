# STATE_TRANSITIONS

## 目的
- 1つのテストが、作成から振り返りまでどう進むかを整理する
- ホーム、テスト、記録の表示分岐を明確にする

## 基本の考え方
このアプリでは、`1テスト = 1サイクル` で考える。

大きな流れはこれ。

1. テストを作る
2. 目標と日程を入れる
3. 学習を進める
4. テスト結果を入れる
5. 振り返る

## Exam の状態
`Exam.status` は次の3つ。

- `planning`
  - まだ準備中
  - テスト作成済みだが、日ごとの学習プランが未確定でもよい
- `active`
  - 学習期間中
  - 今日やることや進捗を見せる対象
- `finished`
  - テスト終了後
  - 結果入力や結果カードの対象

## 状態遷移の全体図

```text
未作成
  ↓ テスト作成
planning
  ↓ 目標点数入力
  ↓ 日程/予定入力
  ↓ 日ごとの学習プラン確定
active
  ↓ 進捗登録を繰り返す
  ↓ テスト終了日を過ぎる
finished（結果未入力）
  ↓ 結果入力
finished（結果入力済み）
  ↓ 次回の引き継ぎ元になる
```

## 状態ごとのホーム表示

### 1. テスト未作成
条件:
- `Exam` が1件もない

ホーム:
- `まずは次のテストを作成しましょう`
- `テストを作成する`

### 2. planning
条件:
- `Exam.status = planning`

ホーム:
- `目標点数と日程を入れましょう`
- `日ごとの学習プランを作る`

補足:
- まだ `今日やること` は出さない

### 3. active
条件:
- `Exam.status = active`

ホーム:
- `2学期中間テストまであと6日`
- `今日やること`
- `進捗サマリー`
- `記録する`

### 4. finished かつ結果未入力
条件:
- `Exam.status = finished`
- `ExamResult` が未登録

ホーム:
- `2学期中間テストが終了しました`
- `今回の点数と勉強時間を記録しましょう`
- `結果を入力する`

### 5. finished かつ結果入力済み
条件:
- `Exam.status = finished`
- `ExamResult` が登録済み

ホーム:
- 次の planning 中または active のテストがあれば、そちらを優先表示
- なければ `次のテストを作成する`

## 各画面の分岐

### テスト一覧
- `planning`
  - `準備中`
- `active`
  - `進行中`
- `finished`
  - `完了`

押した先:
- `planning`
  - 目標点数入力 or 日程/予定入力へ戻れる
- `active`
  - 進行中テスト詳細
- `finished`
  - 結果カード

### 記録
- `active`
  - 今日やった分を記録できる
- `planning`
  - まだ記録対象なし
- `finished`
  - 結果入力を優先するため、進捗記録は閉じるか弱める

## 状態を切り替えるタイミング

### 未作成 -> planning
- テスト作成を完了したとき

### planning -> active
- 次の条件を満たしたとき
  - 目標点数入力済み
  - 日程/予定入力済み
  - 日ごとの学習プランが最低限ある

### active -> finished
- テスト終了日を過ぎたとき

### finished（結果未入力） -> finished（結果入力済み）
- `ExamResult` を保存したとき

## 空状態として持つべきもの

### 手動モード未入力
条件:
- `DailyPlan` がまだ1件もない

表示:
- `まだ学習プランがありません`
- `日付を押して、やる科目と目安時間を置いていきましょう`

### 記録なし
条件:
- `ProgressLog` がまだ1件もない

表示:
- `まだ記録がありません`
- `今日やった分を入れると進捗が更新されます`

### 過去テストなし
条件:
- `finished` の `Exam` がない

表示:
- `まだ振り返れるテストがありません`

## MVPで簡略化してよい点
- `planning` の細かい中間状態を増やさない
- `finished` の中で、結果未入力/入力済みは画面分岐で吸収してもよい
- 複数の active テストは基本想定しない

## 実装前に守ること
- `Exam.status` を UI 文言だけで持たない
- `ExamResult` の有無で `結果未入力/入力済み` を判定できるようにする
- ホームは常に1つの主状態を優先表示する
